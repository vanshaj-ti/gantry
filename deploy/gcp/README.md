# Deploying gantry to GCP

Runs gantry as two always-on Docker containers on one GCE VM:
- `gantry-advance` — ticks `advance --all` against a persistent clone of your target repo.
- `gantry-linear` — receives Linear webhooks, classifies tickets, creates runs.

Set `PROJECT_ID` (and optionally `ZONE` / `VM_NAME`) before running any script —
there is no baked-in cloud project.

## Order of operations

```
01-setup-vm.sh       # create the GCE VM, firewall rule, service account
02-secrets.sh         # create Secret Manager secrets (prompts for real values)
03-deploy.sh          # SSH in, clone target, build image, start both containers
06-ci-setup.md        # one-time: wire GitHub Actions to auto-run 03-deploy.sh
```

Each script is idempotent-ish (safe to re-run) and prints what it's about to
do before doing it. None of them run automatically the first time — read
each one before running.

**After `06-ci-setup.md`'s one-time setup**, every push to `main` that
passes `ci.yml` auto-triggers `.github/workflows/deploy.yml`, which re-runs
`03-deploy.sh` on the VM automatically — see that file for what it does.
Manual `03-deploy.sh` invocation (below) remains available as a fallback/
debugging path, not the normal way to ship a change anymore.

## Prerequisites you provide (not generated by these scripts)

- A fine-grained GitHub PAT scoped to the **target** repo
  (contents: read/write, pull requests: read/write).
- Runner auth for whatever `[agent].runner` the **target project** sets in its
  own `gantry.toml` (Gantry does not hardcode a project or runner):
  - `cursor-sdk` (default for new scaffolds): `CURSOR_API_KEY`
  - Claude Code direct: `ANTHROPIC_API_KEY`, **or**
  - Claude Code via gateway: `ANTHROPIC_BASE_URL` + `ANTHROPIC_AUTH_TOKEN`
  - Codex: `OPENAI_API_KEY` (or whatever env key your `~/.codex/config.toml`
    model_provider uses — pass it through Secret Manager / env-file)
  - `cursor-cli`: host login / CLI auth (not a Secret Manager key)
- The target project's Linear team id (`GANTRY_LINEAR_TEAM_ID`) and a Linear
  personal API key or OAuth token with issue read/write + comment + label
  scopes (`GANTRY_LINEAR_API_KEY`).
- `GANTRY_LINEAR_WEBHOOK_SECRET` — you choose this value now; Linear asks
  for it again when you register the webhook in step 4 (`04-register-webhook.md`),
  so the two must match — Linear does NOT generate this for you.

## Deploy env vars (`03-deploy.sh`)

| Var | Required | Default | Meaning |
|---|---|---|---|
| `TARGET_REPO_URL` | yes | — | Git HTTPS URL of the project gantry builds |
| `GANTRY_REPO_URL` | no | `https://github.com/<org>/gantry.git` | Where to pull gantry source on the VM |
| `TARGET_NAME` | no | basename of repo URL | Used for `/opt/<name>` |
| `TARGET_DIR` | no | `/opt/$TARGET_NAME` | Clone path bind-mounted into containers |
| `BASE_BRANCH` | no | `staging` | Branch to check out / pull |
| `LINEAR_PORT` | no | `8080` | Host port for `gantry-linear` |

Example:

```bash
export TARGET_REPO_URL=https://github.com/acme/my-app.git
export GANTRY_REPO_URL=https://github.com/acme/gantry.git
export BASE_BRANCH=main
sudo -E bash deploy/gcp/03-deploy.sh
```

## After 03-deploy.sh

Follow `04-register-webhook.md` to point a real Linear webhook at the VM's
public endpoint, then `05-verify.md` to run the real Linear-ticket-to-PR test.

## Human-gated stages: answered via Linear comments

Linear is the only human channel for this deployment — no Telegram. When a
run hits a human-gated stage (e.g. `investigation_complete`, awaiting
approval), gantry's comment on the issue is where you reply:
- Reply `approve` (or `yes`/`y`) to advance to the next stage.
- Reply with anything else to send that text back as feedback, which
  resumes the stage with your answer.

This is handled deterministically by `gantry/linear.py`'s
`handle_comment_created` — a plain status lookup + string match (same logic
`gantry listen`'s Telegram path already used), not an agent call. No manual
`docker exec ... gantry approve` needed for normal operation; that remains
available as a fallback if the webhook is down.
